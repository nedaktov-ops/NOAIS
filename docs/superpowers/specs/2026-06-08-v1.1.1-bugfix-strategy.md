## VERIFIED FINDINGS (confirmed by reading source code)

### Fix 1.1 — Hard mode storage area mismatch ✅ CONFIRMED
- `SYNC_KEYS` array (storage-keys.js:44) includes `HARD_MODE_SITES`
- Every consumer reads from `chrome.storage.local` (options.js:149, content.js:134)
- content.js `onStorageChanged` line 307: `if (area !== 'local') return;` — ignores sync entirely
- **Impact:** sync is broken for this key. If sync has stale data, local always wins.
- **Fix:** Move `HARD_MODE_SITES` from `SYNC_KEYS` to `LOCAL_KEYS` in storage-keys.js
- **Cleanup:** Update sync-helper.js line 7 comment (currently says hard-mode is sync)

### Fix 1.2 — Hard mode data format mismatch ✅ CONFIRMED
- storage-keys.js DEFAULTS: `[KEYS.HARD_MODE_SITES]: {}` (object) ✅
- options.js line 67: `currentHardModeSites = []` (array) ❌
- content.js line 148: `const hardMap = (result && result.noais_hard_mode_sites) || {};` (expects object) ✅
- options.js line 162: `Array.isArray(hm) ? hm.filter(...) : []` — reads as array, stores as array
- options.js line 442: `chrome.storage.local.set({ [STORAGE_KEYS.HARD_MODE_SITES]: next })` — stores array
- **Impact:** options stores `["youtube.com"]`, content does `hardMap["youtube.com"]` on array → gets `undefined` → **hard mode NEVER activates**
- **Fix options.js:** change to object format `{hostname: true}` with backwards-compatible array-to-object converter on read
