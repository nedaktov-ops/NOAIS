## Goals

1. **Per-site enable/disable** on a curated list of 7 major platforms (YouTube, Facebook, Instagram, TikTok, X/Twitter, Reddit, LinkedIn), all defaulting to **on**. Plus an "Add custom site" input that lets the user extend the list with any hostname.
2. **Global "Sensitivity" slider** (0–100, default 100) that multiplies the AI-likely score. 0 = ignore all, 50 = half-strength, 100 = raw scores.
3. **Current-site status badge** in the popup: "On this site: ON" (green) or "On this site: OFF" (grey).
4. **Open Settings link** in the popup → opens the new options page in a new browser tab.
5. **Honor per-site setting in the content script** with an early-return when the site is disabled.
6. **Live sync** of settings changes across all open tabs via `chrome.storage.onChanged`.

## In scope

## Storage schema

```json
{
  "noais_enabled": true,
  "noais_global_sensitivity": 100,
  "noais_site_overrides": {
    "example.com": false
  }
}
```

- `noais_enabled` — **EXISTING v0.1 boolean.** Unchanged. Read/write paths stay in `popup.js` for the global toggle.
- `noais_global_sensitivity` — **NEW.** Integer 0–100, default 100. Read by content script (passed to `heuristics.analyzeText`) and options page (slider position).
- `noais_site_overrides` — **NEW.** Object mapping `hostname` (lowercase, no protocol/path) to `boolean` (forced on/off). Only contains user-set overrides; the curated list is **not** stored here (see "Curated list source of truth" below).

**Schema migration:** none required. All new keys; the existing `noais_enabled` boolean is preserved verbatim. A v0.3 user upgrading to v0.4 sees their existing toggle in the popup; new keys default on first read.

**Curated list source of truth:** the 7 curated hosts are a code constant in `core/settings.js`:

```js
const CURATED_HOSTS = [
  'youtube.com', 'facebook.com', 'instagram.com',
  'tiktok.com', 'twitter.com', 'reddit.com', 'linkedin.com',
];
```

At every read, the effective map is `Object.assign({}, ...CURATED_HOSTS.map(h => [h, true]), noais_site_overrides || {})` — the curated defaults are filled in, then user overrides overlay. A future v0.4.x release that adds an 8th curated host automatically appears for all users; no migration step.

**Defensive reads:** every consumer of storage wraps the read in `try/catch` and falls back to defaults. A corrupted or missing key never breaks the extension.

## Hostname matching

```js
function parseHostname(url) {
  // 'https://www.youtube.com/watch?v=abc' -> 'www.youtube.com'
  // 'file:///foo' -> '' (file:// has no hostname)
  // 'chrome://settings' -> '' (unparseable)
  try { return new URL(url).hostname.toLowerCase(); }
  catch { return ''; }
}

function matches(hostname, rule) {
  // Suffix-only match: 'youtube.com' matches 'youtube.com', 'www.youtube.com',
  // 'm.youtube.com'. Does NOT match 'notyoutube.com'.
  if (!hostname || !rule) return false;
  return hostname === rule || hostname.endsWith('.' + rule);
}
```

**Tested cases** (see test plan below): `youtube.com` rule matches `youtube.com` / `www.youtube.com` / `m.youtube.com`; rejects `notyoutube.com` / `youtube.co` / `youtube.com.evil.example`.

**No wildcards in v0.4.** If a user needs `*.example.com`, that is v0.4.x scope.

**The input "Add custom site" field** accepts freeform input, but normalizes:

1. Trim whitespace.
2. If input contains `://`, parse with `new URL`. Otherwise, prepend `https://` and parse.
3. Read `.hostname`, lowercase, strip leading dot.
4. Validate against regex `^[a-z0-9]([a-z0-9-]{0,61}[a-z0-9])?(\.[a-z0-9]([a-z0-9-]{0,61}[a-z0-9])?)+$` and length ≤ 253.
5. Reject with an inline error if validation fails.

## Sensitivity math

```js
// heuristics.analyzeText(text, options = {})
//   options.sensitivity — integer 0–100, default 100
//   If options.sensitivity is provided, raw score is multiplied by
//   (sensitivity / 100) and clamped to [0, 100] before rounding.
```

- 0 → score always 0 (effective "off").
- 50 → raw scores halved.
- 100 → raw scores unchanged (default).
- The clamped score is what the popup displays and what `severityForScore` uses.

The phrase counter is **not** affected by sensitivity. The phrases are exact strings; there's nothing to "tune."

## Options page

**File:** `extension/options/options.html` (with `options.css`, `options.js`).

**Manifest registration:**

```json
"options_ui": {
  "page": "options/options.html",
  "open_in_tab": true
}
```

`open_in_tab: true` is mandatory: the curated list + custom list + slider + saved-toast need more than the 280px popup width.

**Layout (top to bottom):**

1. **Header.** "NOAIS Settings" title, "No Artificial Intelligence Slop" subtitle, "Close" button (closes the tab).
2. **Sensitivity section.** Range input 0–100, default 100. Live label: `0–9 → "Off"`, `10–49 → "Lenient"`, `50–89 → "Default"`, `90–100 → "Strict"`. Saves to `noais_global_sensitivity` on `input` event with 200ms debounce. A "Saved ✓" toast appears for 1.5s after each save.
3. **Per-site list.** Scrollable, max-height 360px. Each row:
   - Hostname (left, `textContent`).
   - Toggle (right) — `<input type="checkbox">` styled as a switch. Writes to `noais_site_overrides[hostname] = checked`. Curated sites show a "curated" badge in grey text.
   - Remove button — only on custom sites. Click → delete key from `noais_site_overrides`, re-render list.
4. **Add custom site row.** Text input + "Add" button. Inline error message area below. On success: input clears, new row appears at the bottom of the list.

**Persistence:** auto-save on every change. No explicit "Save" button.

**Initial load:** read `noais_global_sensitivity` and `noais_site_overrides` from storage. Render the slider position and the list.

## Popup integration

The popup is still 280px wide. Two additions, no resize.

**1. Current-site status line** (below the phrase count, above the toggle):

```
On this site: ON    (green)
On this site: OFF   (grey)
On this site: N/A   (grey, when URL is unparseable)
```

Computed in the popup (not the content script), so it renders even when the content script is unreachable (chrome://, about:blank, file:// with empty hostname). Source: `tab.url` from `chrome.tabs.query({ active: true, currentWindow: true })` → `parseHostname` → `matches(currentHostname, rule)` for each rule → read effective `noais_site_overrides[matchedRule]`.

**2. Open Settings link** (footer, next to the version):

```
v0.4.0  ·  Open Settings
```

Plain underlined text button. On click, calls `chrome.runtime.openOptionsPage()`. Falls back to `chrome.tabs.create({ url: chrome.runtime.getURL('options/options.html') })` if the first throws.

## Content script integration

```js
// At inject time, BEFORE the first scan:
const stored = await chrome.storage.local.get([
  'noais_enabled', 'noais_global_sensitivity', 'noais_site_overrides',
]);
const effective = getEffectiveSettings(stored, location.hostname);
if (!effective.enabled) {
  // Skip the scan. Message handler still mounts but always returns
  // { ok: false, reason: 'site disabled' }.
  mountMessageHandler({ alwaysDisabled: true });
  return;
}
// Proceed with normal analysis, passing effective.sensitivity:
const result = NOAIS_HEURISTICS.analyzeText(text, { sensitivity: effective.sensitivity });
```

**`chrome.storage.onChanged` listener:**

```js
chrome.storage.onChanged.addListener((changes, area) => {
  if (area !== 'local') return;
  if (changes.noais_enabled || changes.noais_site_overrides) {
    // Re-evaluate. If now disabled, switch to "always disabled" mode.
    effective = getEffectiveSettings(...);
  }
  if (changes.noais_global_sensitivity) {
    effective.sensitivity = changes.noais_global_sensitivity.newValue;
  }
});
```

The next popup query will see the updated `effective` because `analyzePage()` reads it dynamically (same pattern v0.3 used for `NOAIS_HEURISTICS`).

## XSS prevention

**Rule:** every user-controllable string is rendered with `textContent` (or `setAttribute`), never `innerHTML`. The audit confirmed this is the v0.3 pattern; v0.4 preserves it.

The options page renders user-added hostnames. The hostname regex rejects HTML syntax, so the input is structurally clean — but the renderer still uses `textContent` defensively.

A specific test case: setting `noais_site_overrides = { "<script>alert(1)</script>": false }` and rendering the options page must display the literal string `<script>alert(1)</script>`, not execute it.
