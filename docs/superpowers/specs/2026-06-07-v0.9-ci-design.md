# v0.9 Design — GitHub Actions CI

**Date:** 2026-06-07
**Author:** NedCode3 + vibe-coder
**Status:** Approved (proceeding)

## 1. Goal

Add continuous integration so every push to `main` and every pull request automatically runs the full NOAIS test suite (199/199 green) on a clean `ubuntu-latest` runner. No more "works on my machine" surprises for contributors or users.

## 2. Non-goals

- No Vitest migration (see §6).
- No release-automation workflow (`gh release` via CI). That's a v1.0 task.
- No code-coverage reporting. The hand-rolled runner doesn't track coverage; adding it would require a new dep.
- No multi-OS matrix (Ubuntu + macOS + Windows). Chrome and Firefox runtimes are Linux-first; if the project needs multi-OS later, it's a one-line matrix expansion.

## 3. Triggers

```yaml
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
  workflow_dispatch:
```

- **push to main** — the integration branch. Catches broken merges.
- **pull_request to main** — every contributor's PR runs the full suite before merge.
- **workflow_dispatch** — manual re-runs from the Actions tab (useful for testing the workflow itself).

## 4. Job shape

```yaml
jobs:
  test:
    name: test-all (Node + headless)
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - checkout
      - apt-get install chromium jq
      - setup-node 18
      - run: make test-all
      - on failure: upload test logs as artifact
```

Total wall-clock estimate: **~45–60 seconds** on a fresh runner. The headless Chromium loads 6 fixtures × 25s timeout each, but in practice each fixture loads in < 2s once Chromium is warm. The job is bounded by a 10-minute timeout for safety.

## 6. Vitest decision (explicit)

The README roadmap calls for "Vitest + CI" in v0.9. **Vitest is intentionally skipped** for v0.9 because:

1. **Zero functional gain.** The hand-rolled `tests/run.js` runner already supports `assert.strictEqual`, async, sub-suites, file discovery, and per-file `PASS/FAIL` output. Our 199 tests use a small subset of Node's `assert` module. Vitest would add 50 MB of `node_modules` and a lockfile for features we don't use.
2. **Zero dep surface.** Vitest pulls in `vite`, `@vitest/*`, `tinypool`, `tinyspy`, `expect`, `chai` adapters, etc. Each of those is a supply-chain risk for a security-sensitive extension. The hand-rolled runner has zero third-party deps.
3. **Faster CI.** `make test-all` takes ~30 s locally. Vitest's startup alone is ~5 s on a fresh runner.
4. **Easy to reverse.** If a future contributor needs Vitest-specific features (snapshot testing, watch mode, parallel worker pool), the migration is mechanical: rewrite each test file's assertion calls from `assert.strictEqual(x, y)` to `expect(x).toBe(y)`, replace `tests/run.js` with a Vitest config, drop `package.json` in. Estimate: half a day.

The decision is recorded here so future contributors don't have to re-litigate it.

## 9. Risks

| Risk | Mitigation |
|---|---|
| `apt-get install chromium` picks up a different version than the local dev box and breaks a selector or quirk | The headless test only depends on Chromium being present; the test asserts the version banner and adapter log. If the apt version diverges, the test will fail loudly and the workflow will be red. Acceptable. |
| `make test-all` flakes on CI due to slow Chromium startup | 10-minute timeout. Each `run_chromium` call has a 25 s `timeout` wrapper. If it still flakes, we move to a per-run retry — but wait for actual evidence first. |
| `actions/checkout@v4` deprecates or breaks | Bump to v5 in a follow-up; this is a routine maintenance task. |

## 10. Out of scope (deferred)

- **Multi-OS matrix.** Add `macos-latest` and `windows-latest` only if a contributor reports a platform-specific issue.
- **Coverage reporting.** Would require an Istanbul/c8 instrumented runner. Skip until v1.1 at earliest.
- **Release automation.** `gh release` from CI is a v1.0 task.
- **Dependency review.** The project has no npm dependencies, so `dependabot.yml` would have nothing to scan. Re-evaluate if Vitest is ever added.
