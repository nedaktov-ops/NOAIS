# v0.7 Design — Instagram + TikTok Adapters

**Date:** 2026-06-07
**Author:** NedCode3 + vibe-coder
**Status:** Draft → Approved (proceeding in parallel)

## 1. Goal

Extend NOAIS platform coverage from 2 adapters (YouTube, Facebook) to 4 by adding **Instagram** and **TikTok** adapters in a single minor release. Both platforms host large volumes of AI-generated comments and captions; both render content as short free-text blocks. Both are top-4 priorities per the README roadmap.

## 2. Non-goals

- No new heuristics. Reuse v0.5's `shortTextMode` thresholds as-is.
- No new storage keys. Storage schema unchanged from v0.4 + v0.5.
- No new UI. Existing soft + hard mode CSS from v0.5 already covers all adapters.
- No mobile-app integration. Web/mobile-web only (already covered by extension).
- No deep frame scan. v0.9+ work.

## 3. Architecture (no change)

The `BaseAdapter` interface introduced in v0.5 is reused verbatim:

```js
{
  id: 'instagram' | 'tiktok',
  match: (hostname) => boolean,
  findElements: (root) => Element[],
  extractText: (el) => string | null,
  decorate: (el, score, count) => void,
  shortTextMode: true
}
```

Both adapters expose their instance via `window.NOAIS_INSTAGRAM_ADAPTER` / `window.NOAIS_TIKTOK_ADAPTER`. `content/content.js` `pickAdapter()` is extended to return the right adapter based on hostname.

## 4. Instagram adapter

### 4.1 Hostname match

```js
match(hostname) {
  return hostname === 'instagram.com'
      || hostname === 'www.instagram.com'
      || hostname === 'm.instagram.com'
      || hostname.endsWith('.instagram.com');  // future-proof
}
```

### 4.2 DOM strategy

Instagram (mobile-web) renders posts and comments as `<article>` elements. Caption / comment text lives inside `[dir="auto"]` descendants (same pattern as Facebook — Instagram and Facebook share the Meta platform's RTL/LTR hint convention).

| Element | Selector | Notes |
|---|---|---|
| Post / comment container | `article` | Matches both feed posts and comment threads. |
| Caption / comment body | `article [dir="auto"]` | First descendant with `>= 30` chars. |
| Header / username / metadata | `article header`, `article time` | Excluded via the 30-char floor. |

### 4.3 Why not data-testid or aria-labels?

Instagram's DOM is heavily versioned and obfuscated. The `[dir="auto"]` convention is stable across both web and m.web since 2019 and is shared with Facebook (already proven in v0.6). The 30-char floor filters out hashtags, mentions, and usernames.

## 5. TikTok adapter

### 5.1 Hostname match

```js
match(hostname) {
  return hostname === 'tiktok.com'
      || hostname === 'www.tiktok.com'
      || hostname === 'm.tiktok.com'
      || hostname.endsWith('.tiktok.com');
}
```

### 5.2 DOM strategy

TikTok (mobile-web) renders comments inside `<div data-e2e="comment-item">` or similar e2e hooks. The hook is the `data-e2e` convention that TikTok uses for internal testing — it's stable across versions.

| Element | Selector | Notes |
|---|---|---|
| Comment container | `[data-e2e="comment-item"]` | One per visible comment. |
| Comment body | `[data-e2e="comment-item"] [data-e2e="comment-text"]` or fallback to first `<p>` / `<span>` >= 30 chars. |
| Username / handle | `[data-e2e="comment-item-username"]` | Excluded via the 30-char floor. |

If `data-e2e` ever disappears in a future TikTok redesign, the fallback (first `<p>`/`<span>` >= 30 chars) keeps the adapter partially functional until v0.8. Document this in CHANGELOG.

### 5.3 Test fixture

`tests/fixtures/test-tiktok.html` — 4 comments + 1 injected:

- Comment 1: "lol" (3 chars) — **skipped**
- Comment 2: "fire" (4 chars) — **skipped** (< 5 words triggers shortTextMode rejection too)
- Comment 3: "This app is so cool, the algorithm is on point" (47 chars) — human
- Comment 4: AI-style longer comment (≥ 50 words) — should score low
- 1 comment injected 100ms after load

### 5.4 Tests

11 tests in `tests/adapters-tiktok.test.js` (mirrors Instagram structure).

```diff

+      "core/adapters/tiktok.js",
       "content/content.js"
     ]
   }]
```

`content/content.js` v0.7.0:

```js
function pickAdapter(host) {
  if (window.NOAIS_INSTAGRAM_ADAPTER?.match(host)) return window.NOAIS_INSTAGRAM_ADAPTER;
  if (window.NOAIS_TIKTOK_ADAPTER?.match(host))      return window.NOAIS_TIKTOK_ADAPTER;
  if (window.NOAIS_FACEBOOK_ADAPTER?.match(host))    return window.NOAIS_FACEBOOK_ADAPTER;
  if (window.NOAIS_YOUTUBE_ADAPTER?.match(host))     return window.NOAIS_YOUTUBE_ADAPTER;
  return null;
}
```

Note: order doesn't matter for correctness, but Instagram and TikTok come first to keep the early-return list alphabetical-ish. Facebook and YouTube remain as v0.6 fallback.

## 8. Branch + worktree strategy (parallel via 2 subagents)

Two independent worktrees so subagents can work in parallel without conflict:

| Worktree | Branch | Owner | Deliverable |
|---|---|---|---|
| `.worktrees/feature-v0.7-instagram` | `feature/v0.7-instagram` | subagent A | `core/adapters/instagram.js` + `tests/adapters-instagram.test.js` + `tests/fixtures/test-instagram.html` |
| `.worktrees/feature-v0.7-tiktok` | `feature/v0.7-tiktok` | subagent B | `core/adapters/tiktok.js` + `tests/adapters-tiktok.test.js` + `tests/fixtures/test-tiktok.html` |

After both subagents complete, I create `.worktrees/feature-v0.7-instagram-tiktok` from `main` and merge both feature branches in, then add the integration glue (manifest, content.js, headless, docs).

## 9. Risks & mitigations

| Risk | Mitigation |
|---|---|
| `data-e2e` selector changes in future TikTok releases | Fallback to first `<p>`/`<span>` >= 30 chars. Documented in CHANGELOG. |
| Instagram DOM doesn't have `article` tags (some inner pages) | Fallback to `main [dir="auto"]` >= 30 chars. Defer to v0.7.1 if needed. |
| Two subagents in parallel conflict on shared files | They're scoped to disjoint files (different adapter + test + fixture). No shared edits. |
| Headless Chromium doesn't reach `m.instagram.com` or `m.tiktok.com` (network blocked) | Use `file://` to local fixture HTML only — same as v0.5/v0.6. |
