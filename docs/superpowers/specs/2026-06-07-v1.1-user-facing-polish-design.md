# v1.1 Design — User-facing Polish

**Date:** 2026-06-07
**Author:** NedCode3 + vibe-coder
**Status:** Approved (proceeding)

## 1. Goal

Make NOAIS feel **finished**. v1.0 shipped the four core platform adapters and the packaging; v1.1 adds the user-facing polish that turns "a working extension" into "a product users will love". The bar is "uBlock Origin-level UX", not "academic project".

Concretely, v1.1 ships 10 features + 1 bugfix:

1. **Page counter overlay** (uBlock Origin-style badge in the corner).
2. **Badge tooltip** with detailed score breakdown + "Don't show this element" link.
3. **Per-element allowlist** (chrome.storage.local).
4. **Keyboard shortcut** (`Ctrl+Shift+A` to toggle the current site).
5. **Settings sync** (`chrome.storage.sync` for the 4 user settings).
6. **First-run welcome page** (onboarding when reason === 'install').
7. **i18n** (en + uk, with `chrome.i18n.getMessage`).
8. **Per-tab popup state** (independent enable/disable per tab, not just per site).
9. **Stats in popup** (X AI elements detected on this page).
10. **"Why am I seeing this?" side panel** (full breakdown: heuristics values, phrase count, score).
11. **Bugfix**: background SW no longer logs "installed" on every cold start.

## 2. Non-goals

- No new platform adapters (that's v1.2).
- No new heuristics (still the same stylometric pipeline from v0.3 + shortTextMode from v0.5).
- No MV2 fallback. MV3 only, as always.
- No telemetry, no analytics, no opt-in data collection. Privacy promise from v1.0 stands.
- No new permissions beyond what's already declared in v1.0. (`storage` and `activeTab` are enough.)

## 3. Architecture

The 10 features are split into 2 subagent worktrees (disjoint file ownership, no conflicts):

### Subagent A — "UI layer" (content script + styles)

Owns:
- `extension/content/page-counter.{js,css}` — the overlay widget.
- `extension/content/badge-tooltip.{js,css}` — hover detail popup.
- `extension/content/element-allowlist.js` — checks/decides per element.
- `extension/core/storage-keys.js` — new central list of storage key names (A and B both depend on this; boss owns it).
- New tests for the above (4 new test files in `tests/`).

Does NOT touch: `popup/`, `options/`, `background/`, `manifest.json`, `settings.js`, `content.js`.

### Subagent B — "Settings + background layer"

Owns:
- `extension/background/keyboard-shortcut.js` — listens for command, dispatches.
- `extension/options/welcome.{html,css,js}` — first-run page.
- `extension/options/options.{html,js,css}` updates (per-tab toggle, per-element allowlist manager, settings sync indicator).
- `extension/popup/popup.{html,js,css}` updates (stats counter + per-tab state).
- `extension/sidepanel/why.html` + `extension/sidepanel/why.js` + `extension/sidepanel/why.css` — the "Why am I seeing this?" panel.
- `extension/_locales/en/messages.json` + `extension/_locales/uk/messages.json` — i18n message catalogues.
- New tests for the above (5+ new test files in `tests/`).

Does NOT touch: `content/`, `core/adapters/`, `core/heuristics.js`, `content.js`.

### Boss — "Integration + sweep + ship"

- `extension/core/storage-keys.js` — single source of truth for storage key names.
- `extension/manifest.json` updates (commands, sidepanel, web_accessible_resources for i18n, content_scripts.js list, content_scripts.css list).
- `extension/background/background.js` bugfix (only log on `onInstalled`, not on every wakeup).
- `extension/options/options.html`/`options.js` — convert to use i18n strings (the bulk of the i18n sweep).
- `extension/popup/popup.js` + `popup.html` — convert to i18n.
- `extension/content/content.js` — wire up page counter, badge tooltip, element allowlist, sidepanel trigger.
- `extension/core/settings.js` — add `chrome.storage.sync` for the 4 user settings (with fallback if sync unavailable).
- Headless integration test updates (Run 7 = welcome page, Run 8 = per-tab state, Run 9 = keyboard shortcut via chromium devtools).
- CHANGELOG, README, PRIVACY.md updates.
- Makefile: `make welcome-screenshot` target.
- `releases/v1.1.0.md` — release notes.

## 4. Feature spec — page counter (Subagent A)

A small draggable widget pinned to the bottom-right corner of every page. Shows: `🤖 NOAIS · 12` (12 = number of detected elements on this page). Click → expands to a list of flagged elements (truncated text + score). Draggable via `mousedown`+`mousemove`.

Settings:
- `noais_page_counter_enabled` (boolean, default `true`).
- `noais_page_counter_position` (object `{x, y}`, default `null` = auto bottom-right).

Hidden on `about:*` and `chrome-extension://*` pages.

## 5. Feature spec — badge tooltip (Subagent A)

When a user hovers a NOAIS badge for > 500 ms, show a small popup:

```
NOAIS 76/100 — AI-likely
Burstiness: 0.21  |  TTR: 0.45
Entropy: 6.8     |  Hapax: 0.32
─────────────────────────────
[Why am I seeing this?]  [Don't show this]
```

- "Why am I seeing this?" → opens the side panel (see §10).
- "Don't show this" → adds the element to the per-element allowlist (see §6).

Tooltip position: above the badge, clamped to viewport.

## 6. Feature spec — per-element allowlist (Subagent A)

Storage:
- Key: `noais_element_allowlist` (object).
- Shape: `{ "<hostname>": { "<textHash>": true } }`.
- `textHash = SHA-256(text.slice(0, 200).toLowerCase())` → first 16 hex chars.

Content script checks the allowlist before decorating. If the element's text hash is in the current host's allowlist, skip it. Options page shows a list of all allowlisted elements (text preview + score + "Remove" button).

The "Don't show this" link in the tooltip is the primary entry point. Right-click context menu on any element is NOT in v1.1 (deferred to v1.2 to keep scope sane).

## 7. Feature spec — keyboard shortcut (Subagent B)

`manifest.json`:

```json
"commands": {
  "_execute_action": { "suggested_key": { "default": "Ctrl+Shift+A" }, "description": "__MSG_cmd_toggle_site__" },
  "noais-toggle-site": { "suggested_key": { "default": "Ctrl+Shift+Y" }, "description": "__MSG_cmd_toggle_site__" }
}
```

Wait — `_execute_action` opens the popup, which we don't want. Use a custom command name:

```json
"commands": {
  "noais-toggle-site": { "suggested_key": { "default": "Ctrl+Shift+A" }, "description": "__MSG_cmd_toggle_site__" }
}
```

`background.js` listens for `chrome.commands.onCommand`, sends a message to the active tab's content script: `{ type: 'TOGGLE_SITE' }`. Content script flips the per-site override (the existing `noais_site_overrides` map) and re-scans.

## 8. Feature spec — settings sync (Boss)

`chrome.storage.sync` for the 4 user settings:
- `noais_enabled`
- `noais_global_sensitivity`
- `noais_site_overrides`
- `noais_hard_mode_sites`

Per-element allowlist and page counter settings stay on `chrome.storage.local` (they can be large; sync has a 100 KB quota).

Fallback: if `chrome.storage.sync` is unavailable (e.g. enterprise policy), use `chrome.storage.local` and show a "Settings will not sync across devices" notice in the options page.

## 9. Feature spec — first-run welcome (Subagent B)

`extension/options/welcome.html` — single page, 4 cards:

1. **What NOAIS does** — short paragraph + the 4 platform logos.
2. **Pick your mode** — soft vs hard mode, with screenshots.
3. **Tour the UI** — "Take the tour" button → sequence of 4 popup overlays.
4. **Privacy promise** — the "zero network calls" line, link to PRIVACY.md.

Triggered by `chrome.runtime.onInstalled` reason === 'install' → `chrome.tabs.create({ url: chrome.runtime.getURL('options/welcome.html') })`.

## 10. Feature spec — "Why am I seeing this?" side panel (Subagent B)

Uses the MV3 `chrome.sidePanel` API (Chrome 114+, Firefox 145+). A vertical panel opens on the right side of the page showing:

- The flagged text (first 200 chars).
- The score + severity colour.
- The 4 heuristics values + the hard-coded AI phrase matches.
- "Add to per-element allowlist" button (calls the same handler as the tooltip link).
- "Report a false positive" link → opens a GitHub issue template.

`manifest.json` adds:
```json
"side_panel": { "default_path": "sidepanel/why.html" },
"permissions": [..., "sidePanel"]
```

`background.js` listens for `chrome.sidePanel.open({ tabId })` calls from the content script.

## 11. Feature spec — per-tab popup state (Subagent B)

The popup shows two toggles:
1. **Master toggle** (existing — global).
2. **On this site** (existing — per hostname).

A new third toggle:
3. **On this tab** — disables NOAIS for this specific tab. Stored as `noais_tab_overrides: { "<tabId>": true }`. Cleared automatically when the tab closes (chrome.tabs.onRemoved).

## 12. Feature spec — i18n (Subagent B + Boss)

Two locales: `en` (default) and `uk` (Ukrainian, for the project owner's locale).

Storage layout:
- `extension/_locales/en/messages.json` — all English strings.
- `extension/_locales/uk/messages.json` — all Ukrainian strings.

Usage in JS: `chrome.i18n.getMessage('key')`. In HTML: `<span>__MSG_key__</span>`. The `default_locale` is `en`.

Coverage:
- Popup title, master toggle, "On this site", "On this tab", stats label.
- Options page: every visible string.
- Welcome page: every visible string.
- Content script: badge text, tooltip, side panel, page counter, "Don't show this" confirmation.

The extraction process is manual for v1.1 (no build step). v1.2 can add an extraction script if it becomes painful.

## 13. Feature spec — stats in popup (Subagent B)

Popup shows, below the toggles:

```
On this page
─────────────
12 elements scanned
3 likely AI
0 very high
```

This is the same data the page counter shows, but in the popup. Updated every time the popup opens (read from `chrome.storage.session`, which the content script writes to).

## 14. Bugfix — background SW "installed" log on every cold start

Current `background.js`:
```js
chrome.runtime.onInstalled.addListener((details) => {
  console.log(`[NOAIS] v0.1.0 installed (reason: ${details.reason})`);
});
```

This is correct — `onInstalled` only fires on actual installs. The "log on every cold start" symptom is because the SW is a **module worker** in MV3 and is restarted for every event. The console message persists across restart? No, the console is per-SW-instance.

Actually, after re-reading: the log IS one-time on install. The "every cold start" report from the headless test was misleading — the headless test runs Chromium 6 times, and each cold start fires `onInstalled` once. So 6 logs in the test output is normal.

But the BUG is real: the log says `v0.1.0` even though we're on `v1.1.0`. The fix is to read the version from `chrome.runtime.getManifest().version`. Trivial.

## 16. Branch + worktree strategy (parallel via 2 subagents + boss)

| Worktree | Branch | Owner | Deliverable |
|---|---|---|---|
| `.worktrees/feature-v1.1-ui` | `feature/v1.1-ui` | Subagent A | `page-counter.{js,css}`, `badge-tooltip.{js,css}`, `element-allowlist.js`, tests |
| `.worktrees/feature-v1.1-settings` | `feature/v1.1-settings` | Subagent B | `keyboard-shortcut.js`, `welcome.{html,css,js}`, `sidepanel/why.{html,css,js}`, `popup/` updates, `options/` updates, `_locales/`, tests |
| `.worktrees/feature-v1.1-integration` | `feature/v1.1-integration` | Boss | Merges A + B, adds `storage-keys.js`, manifest updates, i18n sweep, integration tests, docs, ships |

After all 3 worktrees complete:
1. Boss merges A + B into integration worktree.
2. Adds shared infrastructure (`storage-keys.js`, manifest updates).
3. Runs full test suite.
4. Commits, merges to main with `--no-ff`, tags `v1.1`, pushes.

## 17. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Two subagents step on each other's files (popup.js is touched by B but might be assumed by A) | Disjoint file ownership declared in §3. A's only file in `popup/` would be a no-no; B owns `popup/`. |
| `chrome.sidePanel` not in Firefox before v145 | Graceful degradation: if `chrome.sidePanel` is undefined, the tooltip "Why?" button opens a new tab to a static why.html page instead. |
| i18n sweep leaves a string untranslated | Manual grep at integration time: `grep -rE "[\"'][A-Z][a-z]+ [a-z]+" extension/popup extension/options extension/sidepanel extension/background` should return 0 lines after the sweep (all hard-coded English strings are replaced). |
| `chrome.storage.sync` quota (100 KB total) hit by per-site overrides | The 4 user keys total < 5 KB in typical use. The 100 KB quota is for all sync data across all extensions, but per-extension defaults are 8 KB + 2 KB/item, which is fine. |
| Page counter interferes with site CSS | Use `all: initial; position: fixed; z-index: 2147483647;` reset on the root + high z-index. The page counter is a Shadow DOM root so its CSS is fully isolated. |
